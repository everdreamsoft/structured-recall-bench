A note on *-planned variants

We experimented with giving mem0 the same LLM query planner that sandra-structured uses (mem0-planned, smoke test only). Composite dropped from 0.262 (raw) to 0.174 (planned). Why: a planner generates targeted semantic queries — but a top-K semantic index, by construction, cannot return the exhaustive filtered set a structured question demands. The more precise the query, the more obvious the ceiling. We did not build zep-planned or supermemory-planned for the same reason: the bottleneck is the retrieval backend, not the query formulation. Adding a planner to a top-K store is architecturally incapable of closing the gap this benchmark exposes.
